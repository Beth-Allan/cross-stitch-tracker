<objective>
Show "already added" indicator for project supplies in the search-to-add dropdown instead of hiding them.

Purpose: Currently, searching for a thread/bead/specialty already in the project returns "No matches" because `existingIds` are filtered out entirely. This is confusing -- the user doesn't know if the item doesn't exist or is already added. Instead, show those items greyed out with an "Already added" label, making it clear the item exists but is already part of the project.

Output: Updated `search-to-add.tsx` component + colocated tests.
</objective>

    **Step 1 — Change filtering logic (line ~125):**

    Replace:
    ```typescript
    const filtered = items.filter((item) => !existingSet.has(getItemId(item))).slice(0, 8);
    ```

    With a partition approach that shows ALL items but marks which are already added. Limit total to 8 items, but sort so addable items appear first, then already-added:
    ```typescript
    const addable = items.filter((item) => !existingSet.has(getItemId(item)));
    const alreadyAdded = items.filter((item) => existingSet.has(getItemId(item)));
    const displayItems = [...addable, ...alreadyAdded].slice(0, 8);
    ```

    **Step 2 — Update keyboard navigation:**

    The `highlightIndex` should only land on addable items. Modify `handleKeyDown`:
    - ArrowDown: find next index where item is NOT in existingSet
    - ArrowUp: find previous index where item is NOT in existingSet
    - Enter: only act if `displayItems[highlightIndex]` is addable

    Also update initial `highlightIndex` in the fetch effect to point to the first addable item (should be 0 since addable items come first, but be defensive).

    **Step 3 — Update the "No matches" condition:**

    Change the empty state check from `filtered.length === 0` to `items.length === 0` (truly no search results at all vs all results being already-added).

    **Step 4 — Update the render for each item:**

    For already-added items:
    - Add `disabled` attribute to the button
    - Add `opacity-50` and `cursor-default` classes
    - Remove `hover:bg-muted` class (no hover effect)

    For addable items:
    - Keep existing behavior unchanged

    Use `existingSet.has(getItemId(item))` to determine which rendering path.

    All new styles must use semantic tokens (text-muted-foreground, bg-muted, etc.) per project conventions. No hardcoded color scales.
